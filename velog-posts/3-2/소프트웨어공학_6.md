<h1 id="🔥-요구분석-단계-대표적-실수들">🔥 요구분석 단계 대표적 실수들</h1>
<h2 id="1-모호한-요구사항-정의">1. 모호한 요구사항 정의</h2>
<ul>
<li>요구사항이 불분명하거나 모호한 표현으로 인한 오류</li>
<li><h3 id="span-stylecolorindianredsmart원칙span"><span style="color: indianred;">SMART원칙</span></h3>
Specfic-Mesurable-Acievable-Relevent-Time bound<h2 id="2-요구사항의-누락">2. 요구사항의 누락</h2>
</li>
<li>중요한 기능 또는 비기능적 요구사항이 누락</li>
<li>체크리스트를 활용하여 각 요구사항 수집을 세밀히 체크</li>
<li>워크샵 및 검토화(Review) 적극 활용<h2 id="3-커뮤니케이션-오류">3. 커뮤니케이션 오류</h2>
</li>
<li>고객-개발자 간 의사소통 및 이해 부족으로 인한 오류</li>
<li>프로토타입 활용 및 정기적 미팅(고객 피드백 적그 도출)</li>
</ul>
<hr />
<h1 id="📌-요구분석-실수가-미치는-영향">📌 요구분석 실수가 미치는 영향</h1>
<h2 id="1-프로젝트-일정-지연">1. 프로젝트 일정 지연</h2>
<ul>
<li>요구사항 변경 및 수정으로 추가 개발시간 필요</li>
<li>프로젝트 전체 일정의 지연으로 연결<h2 id="2-프로젝트-비용-증가">2. 프로젝트 비용 증가</h2>
</li>
<li>새로운 기능 개발에 따른 개발비용 증가</li>
<li>기존 기능 수정에 따른 개발비용 증가<h2 id="3-품질-저하-가능성-증가">3. 품질 저하 가능성 증가</h2>
</li>
<li>요구사항 변경에 따른 기능 변화로 테스트와 검증 부족</li>
<li>전체적인 시스템 품질 저하로 연결</li>
</ul>
<hr />
<h1 id="📌-실무-최소화---우선순위-적용">📌 실무 최소화 - 우선순위 적용</h1>
<h2 id="1-우선순위의중요성">1. 우선순위의중요성</h2>
<ul>
<li>요구분석 실수 대부분 불명확한 기능을 전부 포함하려는 욕심</li>
<li>한정된 자원 - 필수기능(Must Have) 범우 확정 - 성공요인</li>
<li>우선순위 낮은 요군ㄴ 후에 변경/수정 요인이 많음<h2 id="2-요구사항-우선순위-기법">2. 요구사항 우선순위 기법</h2>
</li>
<li>애자일방식에서 요구사항 우선순위 설정하는 대표적 방법</li>
<li><h3 id="span-stylecolorindianredmoscowspan기법-적용"><span style="color: indianred;">MoSCoW</span>기법 적용</h3>
Must Have-Should Have-Could Have-Won't Have<h2 id="3-작은-변화를-자주-반영하는-전략">3. 작은 변화를 자주 반영하는 전략</h2>
</li>
<li>큰 단위로 변경 대신, 작은 단위로 자주 변경하는 방법</li>
<li>테스트와 검증이 용이 하며, 문제 발생시, 빠른 수정과 대처</li>
</ul>

<h1 id="🔥-요구사항-변경의-주요-원인">🔥 요구사항 변경의 주요 원인</h1>
<h2 id="1-비지니스-변화">1. 비지니스 변화</h2>
<ul>
<li>기업 목표 또는 외부 환경이 변화화여 요구사항 수정되는 경우</li>
<li>예) 경쟁사 새 기능 또는 제품 출시<h2 id="2-기술적-제한">2. 기술적 제한</h2>
</li>
<li>개발 중 기술적 문제나 한계로 기존 요구사항 수정하는 상황</li>
<li>개발 중 새로운 기술의 발견으로 요구사항 수정하는 상황<h2 id="3-법적-또는-규제-요구사항의-변경">3. 법적 또는 규제 요구사항의 변경</h2>
</li>
<li>국가 법률 또는 산업규제에 따른 환경 요건 변화</li>
<li>기술 자료 또는 기술관련 환경의 변화</li>
</ul>
<hr />
<h1 id="📌-변경관리-프로세스">📌 변경관리 프로세스</h1>
<h2 id="1-변경-영향-분석">1. 변경 영향 분석</h2>
<ul>
<li>변경 사항이 시스템에 미치는 영향을 평가</li>
<li>개발팀이 언제, 어떻게, 어느정도 분량으로 반영할지 결정<h2 id="2-리스크-분석">2. 리스크 분석</h2>
</li>
<li>변경 사항이 시스템 다른 부분에 영향을 미치는지 검토<ul>
<li><span style="color: indianred;">일정, 성능, 보안, 안정성</span>에 미치는 영향 세밀히 검토 <h2 id="3-연결성-관리">3. 연결성 관리</h2>
</li>
</ul>
</li>
<li>변경 사항과 구현 코드, 테스트 간 연결성 관리</li>
<li>요구 추적성 관리를 통해 <span style="color: indianred;">요구사항-코드-테스트</span> 간 연결 파악</li>
</ul>

<h1 id="📌-요구사항-불확실성-최소화-방법">📌 요구사항 불확실성 최소화 방법</h1>
<h2 id="1-프로토타입">1. 프로토타입</h2>
<ul>
<li>고객 요구기능을 프로토타입으로 시각화</li>
<li>이해관계자(고객, 개발자) 상호 확인 및 보안<h2 id="2-요구사항-워크샵">2. 요구사항 워크샵</h2>
</li>
<li>정기적 미팅 및 워크샵 적극 활용</li>
<li>이해관계자 간 의사소통 및 상호 확인/검증 - 검토화(Review)<h2 id="3-점진적-개발aglie">3. 점진적 개발(Aglie)</h2>
</li>
<li>단계별 개발양식 채택</li>
<li>요구사항 빠르게 확인 및 반영, 주기적 검토 진행</li>
</ul>

<ul>
<li>사용자 연구(User Research)를 통한 사용자 문제 이해</li>
<li>다양한 공감 기법 활용<ul>
<li>관찰, 인터뷰, 설문조사, 워크샵 등<h2 id="예시">예시</h2>
</li>
</ul>
</li>
<li>현 고객 웹사이트 기능 검토</li>
<li>고객 불편사항 및 원하는 기능 파악 (고객의 입장에서)</li>
<li>실제 구현할 기능 요구사항 정리</li>
<li>불필요한 기능 또는 서비스 제외 (고객 설득)</li>
<li>반복 의견교환 및 확정 (검토화)</li>

<h1 id="📌-문제정의-단계-define---요구사항-구체화">📌 문제정의 단계 (Define) - 요구사항 구체화</h1>

<li>핵심 요구사항만 도출 - 불필요 요구사항 제외 및 설득</li>
<li>최종 개발범위 확정<h2 id="예시-1">예시</h2>
</li>
<li>현 고객 웹사이트 기능 중 검색 기능 불만 확인</li>
<li>문제점 식별 및 수정 방안 검토</li>
<li>검색 기능 요구사항 혐의 - 필수 요구사항(Must Have)추출</li>
<li>최정 기능 정의</li>
</ul>

<h1 id="📌아이디어-도출-ideation---창의적-해결책-탐색">📌아이디어 도출 (Ideation) - 창의적 해결책 탐색</h1>
<h2 id="개요-2">개요</h2>
<ul>
<li>다양한 해결책을 제시</li>
<li>기능적, 비기능적 요구를 구체화</li>
<li>브레인스토밍 방법이 효율적</li>
<li>현실적으로 가능한 요구를 우선순위화<h2 id="예시-2">예시</h2>
</li>
<li>현 고객 웹사이트 기능 중 검색 대체기능 제시</li>
<li>한 가지만 아니라 다양한 기능과 방법을 고려</li>
<li>기술 검토하여 다양한 의견 중, 현실 가ㅡㅇ한 방법으로 축약</li>
<li>우선순위화 - 기간/기술/비용 고려</li>
</ul>
